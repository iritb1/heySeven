# heySeven - Casino Concierge AI Agent

A conversational AI agent that answers guest questions about **Twin Arrows Navajo Casino Resort** — its restaurants, entertainment, amenities, rooms, promotions, and anything a guest might ask before or during a visit.

Built with **LangGraph**, **FastAPI**, and **OpenAI GPT-4o**.

---

## Architecture

```
Guest Question
      |
      v
  guard_node          Classifies: is this about the property?
      |
  in_scope?
   /      \
  YES      NO
  |         |
  v         v
context    refusal_node     Fixed polite refusal
node             |
  |              END
  v
answer_node     LLM generates response with property context
  |
 END
```

The agent uses a **LangGraph StateGraph** with four nodes:

| Node | Purpose |
|------|---------|
| **guard_node** | Classifies if the question is property-related. Defaults to refusal on error. |
| **context_node** | Injects property data as a SystemMessage (once per thread to avoid duplication). |
| **answer_node** | Generates the final answer using the full conversation history. |
| **refusal_node** | Returns a polite refusal for off-topic questions. No LLM call. |

**Multi-turn conversations** are supported via `MemorySaver` checkpointer and `thread_id`. The first request generates a `thread_id` and returns it — subsequent requests with the same `thread_id` continue the conversation with full context.

## Quick Start

### Prerequisites
- Python 3.12+
- An OpenAI API key

### Local Setup

```bash
# Clone and enter the project
cd heySeven

# Create virtual environment
python -m venv .venv
source .venv/bin/activate

# Install dependencies
pip install -r requirements.txt

# Configure environment
cp .env.example .env
# Edit .env and add your OPENAI_API_KEY

# Generate property data
PYTHONPATH=. python ingestion/ingest.py

# Start the server
PYTHONPATH=. uvicorn api.main:app --reload

# Open in browser
# UI:      http://localhost:8000
# Swagger: http://localhost:8000/docs
```

### Docker

```bash
cp .env.example .env
# Edit .env and add your OPENAI_API_KEY

docker compose up --build
```

The container runs ingestion automatically on startup, then starts the API on port 8000.

## API

### `GET /health`
Liveness check.
```json
{"status": "ok"}
```

### `POST /chat`
Send a guest question. Returns the agent's answer and a `thread_id` for multi-turn conversations.

**First message** (new conversation):
```json
// Request
{"message": "What restaurants do you have?"}

// Response
{
  "answer": "We have four restaurants: Zenith Steakhouse (fine dining)...",
  "thread_id": "a1b2c3d4-e5f6-..."
}
```

**Follow-up** (same conversation):
```json
// Request
{"message": "What's the dress code for the first one?", "thread_id": "a1b2c3d4-e5f6-..."}

// Response
{
  "answer": "Zenith Steakhouse has a smart casual dress code...",
  "thread_id": "a1b2c3d4-e5f6-..."
}
```

## Design Decisions & Trade-offs

### LLM-generated property data
Property data is generated by GPT-4o from its training knowledge rather than scraped from the website. **Trade-off**: faster and more maintainable (no brittle HTML parsing), but data may not reflect real-time changes (e.g., current promotions). In production, I'd combine this with a live data source or periodic re-generation. For a larger-scale system with multiple properties, MCP servers could expose scraping and data retrieval as tools the agent calls at runtime — enabling live, per-property data access without hardcoding each source.

### Guard node for scope control
A dedicated classification step before answering ensures the agent stays on-topic. The guard defaults to **refusal on error** — a safe-by-default approach. This adds one extra LLM call per request but prevents hallucination on unrelated topics.

### In-memory conversation storage
`MemorySaver` stores conversation history in process memory. 
**Trade-off**: simple and fast, but conversations are lost on restart and can't be shared across multiple server instances. In production, I'd swap for `PostgresSaver` for persistence and horizontal scaling.

### Singleton LLM client
The OpenAI client is instantiated once (lazy, on first use) and reused across all requests. This follows OpenAI's recommended pattern and avoids creating new HTTP connections per request.

### Time-aware concierge
The system prompt includes the current date/time in MST (Arizona timezone), enabling the agent to answer "what's open now?" questions by comparing against restaurant hours.

## What I'd improve with more time

- **Persistent conversation storage** — Swap `MemorySaver` for `PostgresSaver` so conversations survive restarts and work across multiple server instances
- **RAG with Vector DB** — Currently the entire property data (~4KB) fits in the system prompt. For larger datasets (full menus, event calendars, reviews), I'd embed chunks into a vector DB and retrieve only relevant context per question — reducing token usage and scaling to richer property data
- **Streaming responses** — Use LangGraph's `astream` for real-time token-by-token delivery to the UI instead of waiting for the full response
- **Live data ingestion** — Scrape the actual website or integrate with a property management API for real-time accuracy
- **LangSmith tracing** — Already configured in `.env.example`. Adding an API key enables full observability — trace every LLM call, monitor token costs, and debug guard misclassifications without any code changes
- **Rate limiting** — Protect against abuse on the `/chat` endpoint
- **Authentication** — API key or JWT to protect against unauthorized usage and OpenAI cost abuse
